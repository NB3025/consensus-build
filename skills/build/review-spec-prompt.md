# 기능정의서 리뷰 프롬프트

Agent Teams 3명 각각에게 전달할 검토 지시.

기능정의서 파일을 Read 도구로 읽은 후, 아래 지시에 따라 검토하라.

파일 경로: {Phase 1에서 저장한 파일 경로}

아래 문서의 모든 섹션을 빠짐없이 검토할 것.

일반적인 품질(완성도, 실현가능성, 리스크, 보안 영향)과 함께,
다음 5가지 일관성 관점을 점검할 것:
1. 수치/데이터 모순 — 같은 지표가 문서 내 다른 위치에서 다른 값으로 등장
2. 타임라인 모순 — 일정·마일스톤·순서 역행
3. 용어/정의 모순 — 같은 개념에 다른 이름, 또는 같은 이름에 다른 의미
4. 전제/규칙 모순 — 앞에서 정한 원칙을 뒤에서 위반
5. 논리 모순 — 분석/근거와 결론/전략 불일치

### 검토 순서
5가지 관점을 한꺼번에 보지 말 것.
하나씩 순서대로: 수치/데이터 → 타임라인 → 용어/정의 → 전제/규칙 → 논리모순.
각 관점마다 문서 전체를 처음부터 끝까지 다시 읽을 것.
일반 품질 검토는 5가지 일관성 점검을 모두 마친 후 마지막에 수행할 것.

### 모순 검토 절차
1단계(추출): 문서 전체에서 수치, 날짜, 정의, 원칙, 논리적 주장을 추출
2단계(쌍 비교): 추출한 항목들을 쌍으로 교차 비교하여 모순 여부 판별
3단계(근거 체인): 모순 발견 시 양쪽 원문을 정확히 인용하고 근거 체인 구성

### 출력 형식

발견한 이슈마다:

#### ISS-{번호}: {이슈 제목}
- **유형**: 일반 | 수치모순 | 타임라인모순 | 용어모순 | 전제모순 | 논리모순
- **심각도**: CRITICAL | MAJOR | MINOR
- **확신도**: 1-10 (이 이슈가 실제 문제일 확신도)
- **근거 강도**: 1-5 (근거의 구체성·명확성)
- **추론 과정**: 왜 이것이 문제/모순인지
- **근거**:
  📍 '{문장 A}' (섹션 X)
  📍 '{문장 B}' (섹션 Y)
- **결론**: 어떤 유형의 문제이며 어떤 영향이 있는지
- **제안**: 수정 방향

Good 예시:
- 확신도 9, 근거강도 5:
  📍 '3장: MAU 50만 명' (섹션 3.2)
  📍 '5장: MAU 30만 명 기준 수익 모델' (섹션 5.1)
  → 추론: 동일 지표가 20만 차이. 수익 모델 전체에 영향.

Bad 예시:
- 확신도 8, 근거강도 2:
  '전반적으로 수치가 안 맞는 느낌'
  → 원문 인용 없음. 이 경우 확신도를 3 이하로 매길 것.

### 완성도 검토 시 추가 기준

범위 제외(scope exclusion) 항목이 있다면 다음을 점검할 것:
- 제외 근거가 기술적으로 타당한가
- 제외된 항목이 포함된 항목의 정합성이나 효과를 훼손하지 않는가
- 해당 기능이 준수해야 하는 표준, 프로토콜, API 계약, 또는 사용자 기대 중 제외된 것으로 인해 포함된 수정의 실효성이 훼손되지 않는가
- 제외 항목의 후속 대응 계획이 명시되었는가

동작 변경이 수반되는 경우 다음을 점검할 것:
- 기존 사용자/호출자에 대한 하위 호환성 영향이 식별되었는가
- 영향이 있다면 완화 방안 또는 마이그레이션 경로가 기술되었는가

이 기준은 소스 코드를 읽지 않고 문서 내 정보만으로 판단한다.

⚠️ 규칙:
- 원문 인용은 문서에서 정확히 복사할 것. 한 글자라도 다르면 안 됨.
- 정확한 인용을 못 하겠으면 확신도를 3 이하로 매길 것.
- 모순 유형은 반드시 모순되는 양쪽 문장 2개를 인용할 것.
- 일반 유형도 근거가 되는 원문을 인용할 것.
- 문서에 없는 문장을 인용으로 꾸며내지 말 것.
- 이슈 0건이면 '이슈 없음'으로 보고할 것. 억지로 만들지 말 것.

⚠️ 다시 한번: 원문 인용은 문서 원문 그대로 복사. 변형·요약·의역 금지.
