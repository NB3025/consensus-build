# 구현 계획 리뷰 프롬프트

Agent Teams 3명 각각에게 전달할 검토 지시.

구현 계획 파일을 Read 도구로 읽은 후, 아래 지시에 따라 검토하라.

파일 경로: {Phase 3에서 저장한 구현 계획 파일 경로}

아래 구현 계획의 모든 태스크를 빠짐없이 검토할 것.

일반적인 품질(완성도, 실현가능성, 누락된 태스크)과 함께,

"일반 품질" 검토 시 다음을 추가 확인할 것:
- 변경이 기존 공개 API/인터페이스의 하위 호환성을 깨뜨리는가 (breaking change)
- 변경에 보안 영향이 있는가 (입력 검증, 인증/인가, 민감 데이터 경로)
- 에러 발생 시 전파 경로와 호출자의 처리 방안이 고려되었는가

다음 7가지 일관성 관점을 점검할 것:
1. 설정값/리소스명 불일치 — 같은 리소스·변수·설정이 태스크 간 다른 이름이나 값으로 등장
2. 의존성/순서 모순 — 태스크 A가 태스크 B의 산출물을 사용하는데 B가 뒤에 있음, 또는 순환 의존
3. 명칭/용어 불일치 — 같은 리소스를 다른 이름으로 참조, 또는 같은 이름이 다른 리소스를 지칭
4. 요구사항/구현 불일치 — Requirements에서 요구한 것과 태스크 구현 내용이 다름, 또는 요구사항 누락
5. 논리 모순 — 태스크 간 설계 결정이 충돌 (예: 태스크 3에서 퍼블릭 접근 허용인데 태스크 5에서 차단). 행위 변경(기능 추가/수정)과 구조 변경(코드 정리/재구조화)이 동일 태스크에 혼재되어 있는 경우도 논리 모순으로 판별
6. 프로젝트 관행 일관성 — 구현 방식(검증 패턴, 에러 처리, 유틸리티 사용)이 대상 코드베이스의 기존 관행과 일치하는가. 동일한 문제를 해결하는 기존 코드가 있다면 그 패턴을 참조하거나 일탈 근거를 명시했는가. 기존 코드에서 확립된 최적화 패턴이 있다면 이를 따르는가
7. 테스트 전략 체계성 — 요구사항에서 도출된 edge case가 테스트 계획에 빠짐없이 매핑되었는가. 유사한 입력 변형을 개별 테스트로 나열하지 않고 매개변수화 또는 데이터 주도 방식으로 구조화할 수 있는 기회가 있는가. 기능정의서의 WHEN/THEN 쌍과 테스트의 1:1 추적이 가능한가

### 검토 순서
7가지 관점을 한꺼번에 보지 말 것.
하나씩 순서대로: 설정값/리소스명 → 의존성/순서 → 명칭/용어 → 요구사항/구현 → 논리모순 → 프로젝트관행 → 테스트전략.
각 관점마다 계획 전체를 처음부터 끝까지 다시 읽을 것.
일반 품질 검토는 7가지 일관성 점검을 모두 마친 후 마지막에 수행할 것.

### 모순 검토 절차
1단계(추출): 계획 전체에서 리소스명, 변수명, 설정값, 의존 관계, 요구사항 매핑, 사용된 API/패턴, 테스트 설계를 추출
2단계(쌍 비교): 추출한 항목들을 태스크 간 교차 비교하여 불일치 여부 판별
3단계(근거 체인): 불일치 발견 시 양쪽 원문을 정확히 인용하고 근거 체인 구성

### 출력 형식

발견한 이슈마다:

#### ISS-{번호}: {이슈 제목}
- **유형**: 일반 | 설정값불일치 | 의존성모순 | 명칭불일치 | 요구사항불일치 | 논리모순 | 관행불일치 | 테스트누락
- **심각도**: CRITICAL | MAJOR | MINOR
- **확신도**: 1-10 (이 이슈가 실제 문제일 확신도)
- **근거 강도**: 1-5 (근거의 구체성·명확성)
- **추론 과정**: 왜 이것이 문제/불일치인지
- **근거**:
  📍 '{태스크 A 원문}' (태스크 X)
  📍 '{태스크 B 원문}' (태스크 Y)
- **결론**: 어떤 유형의 문제이며 구현 시 어떤 영향이 있는지
- **제안**: 수정 방향

Good 예시:
- 확신도 9, 근거강도 5:
  📍 '태스크 2에서 생성하는 항목의 이름이 X' (태스크 2)
  📍 '태스크 5에서 참조하는 항목의 이름이 Y' (태스크 5)
  → 추론: 같은 항목인데 이름이 다름. 구현 시 태스크 5가 존재하지 않는 대상을 참조하게 됨.

Bad 예시:
- 확신도 8, 근거강도 2:
  '전반적으로 태스크 간 연결이 느슨한 느낌'
  → 원문 인용 없음. 이 경우 확신도를 3 이하로 매길 것.

⚠️ 규칙:
- 원문 인용은 계획 문서에서 정확히 복사할 것. 한 글자라도 다르면 안 됨.
- 정확한 인용을 못 하겠으면 확신도를 3 이하로 매길 것.
- 불일치 유형은 반드시 불일치하는 양쪽 원문 2개를 인용할 것.
- 일반 유형도 근거가 되는 원문을 인용할 것.
- 스타일 취향(태스크 순서 선호, 네이밍 컨벤션 취향)을 이슈로 보고하지 말 것.
- 프레임워크/도구가 자동으로 해결하는 것을 이슈로 올리지 말 것. 단, 자동 해결이 확실하지 않으면 보고할 것.
- 문서에 없는 내용을 인용으로 꾸며내지 말 것.
- 이슈 0건이면 '이슈 없음'으로 보고할 것. 억지로 만들지 말 것.
- 관점 6-7에서는 대상 코드베이스의 기존 패턴을 확인하기 위해 소스 코드를 참조할 수 있다. 단, 구현 계획 문서가 주 검토 대상이며, 소스 코드는 계획의 타당성을 확인하기 위한 보조 참조로만 사용한다.
- 관행 불일치 지적 시, 기존 코드에서 해당 관행이 사용된 위치를 가능하면 인용할 것.

⚠️ 다시 한번: 원문 인용은 계획 문서 원문 그대로 복사. 변형·요약·의역 금지.
